# Subgraph Migration: Outcome-Bound Payout Authorization (Multisig v2)

Audience: TheGraph / subgraph engineers. This is the companion to the contract
migration ("Multisig v2") and to SDK **v3.0.0**, whose GraphQL queries
(`src/subgraph/queries.ts`) already request the new fields described below.
**Until the subgraph ships this schema and is reindexed, SDK 3.x subgraph
queries will fail** — coordinate the deployments.

The v2 ABIs referenced here are checked into this repo and are the source of
truth for the subgraph's `abis/`:

- `src/contract/PalindromePay.json` (escrow)
- `src/contract/PalindromePayWallet.json` (per-escrow wallet)

Outcome values used throughout (the contract's `State` enum):

| outcome | value | meaning |
|---|---|---|
| `COMPLETE` | 3 | release to seller (minus fees) |
| `REFUNDED` | 4 | full refund to buyer (dispute) |
| `CANCELED` | 5 | full refund to buyer (cancel) |

## 1. Changed events (breaking — new topics)

The three signature events gained a trailing `outcome` param:

```solidity
event SellerWalletSigAttached(uint256 indexed escrowId, bytes sellerSig, State outcome);
event BuyerWalletSigAttached(uint256 indexed escrowId, bytes buyerSig, State outcome);
event ArbiterWalletSigAttached(uint256 indexed escrowId, bytes arbiterSig, State outcome);
```

Because the parameter list changed, the event signature hash (topic0) changed.
Handlers subscribed to the old signatures silently stop matching — update the
`eventHandlers` entries in `subgraph.yaml` to the new signatures.

**Upsert semantics.** A participant's stored signature now reflects the outcome
they *currently* consent to, and later actions **overwrite** it:

- create/deposit/accept/confirmDelivery attach a `COMPLETE` (3) signature;
- `requestCancel` re-attaches the caller's signature with `CANCELED` (5);
- `refundAfterDisputeTimeout` re-attaches the buyer's with `REFUNDED` (4);
- `submitArbiterDecision` attaches the arbiter's with the ruling (3 or 4).

So the handlers must **update** the stored sig + outcome on every event, never
insert-once/skip-if-present.

## 2. New events

```solidity
// PalindromePay (escrow): arbiter assigned later by mutual consent (setArbiterSigned)
event ArbiterSet(uint256 indexed escrowId, address indexed arbiter);

// PalindromePayWallet (per-escrow wallet, same data-source template as Withdrawn):
// emitted only when an arbiter fee is actually paid out
event ArbiterPaid(uint256 indexed escrowId, address indexed arbiter, uint256 amount);
```

- `ArbiterSet` handler: update `Escrow.arbiter` **and refresh `Escrow.arbiterFeeBps`**
  (the fee is set in the same transaction; read it via a bound `getEscrow` call).
- `ArbiterPaid` handler: surface the paid fee (e.g. `Escrow.arbiterPaidAmount` or an
  event entity) — the SDK does not query it yet, model as you prefer.

## 3. Changed views / struct (regenerate bindings)

`getEscrow(uint256)` returns the `EscrowDeal` tuple with **two new fields**:

| # | field | type | note |
|---|---|---|---|
| 8 | `maturityDuration` | `uint256` | NEW — between `maturityTime` and `disputeStartTime` |
| 14 | `arbiterFeeBps` | `uint16` | NEW — after `tokenDecimals` |

Full order: `token, buyer, seller, arbiter, wallet, amount, depositTime,
maturityTime, maturityDuration, disputeStartTime, state, buyerCancelRequested,
sellerCancelRequested, tokenDecimals, arbiterFeeBps, sellerWalletSig,
buyerWalletSig, arbiterWalletSig`.

Any handler decoding this tuple positionally must add the two fields; bindings
generated by `graph codegen` from the new ABI handle it automatically. The
wallet views also changed (`getAuthorizationDigest(address,uint8)`,
`getValidSignatureCount(uint8)`, `isSignatureValid(address,uint8)`) — only
relevant if a handler calls them.

`createEscrow` / `createEscrowAndDeposit` gained a `uint16 arbiterFeeBps`
parameter (inserted after `arbiter`) — only relevant if you use `callHandlers`.

## 4. `deposit()` re-anchors `maturityTime` (behavior change)

`deposit()` now restarts the delivery window at deposit time:
`maturityTime = depositTime + maturityDuration`. The `PaymentDeposited` event
payload is unchanged (`escrowId, buyer, amount`), so the handler **must refresh
`Escrow.maturityTime`** — either recompute from the stored `maturityDuration`
and the block timestamp, or (recommended, exact) re-read via the bound
`getEscrow(escrowId)` call.

## 5. Schema additions (names are the contract with SDK v3)

SDK `3.x` queries these exact field names on the `Escrow` entity — if you pick
different names, `src/subgraph/queries.ts` and `src/types/escrow.ts` in this
repo must be changed to match:

```graphql
type Escrow @entity {
  # ... existing fields ...
  sellerWalletSigOutcome: Int      # State the seller sig authorizes (3/4/5)
  buyerWalletSigOutcome: Int       # State the buyer sig authorizes (3/4/5)
  arbiterWalletSigOutcome: Int     # State the arbiter sig authorizes (3/4)
  arbiterFeeBps: Int               # arbiter fee in bps (0-2000)
  maturityDuration: BigInt         # delivery window in seconds
}
```

## 6. Handler → field mapping

| Handler (event) | Fields to set/update |
|---|---|
| `EscrowCreated` | existing fields + `arbiterFeeBps`, `maturityDuration` (not in the event — read via `getEscrow`) |
| `SellerWalletSigAttached` | `sellerWalletSig`, `sellerWalletSigOutcome` (upsert) |
| `BuyerWalletSigAttached` | `buyerWalletSig`, `buyerWalletSigOutcome` (upsert) |
| `ArbiterWalletSigAttached` | `arbiterWalletSig`, `arbiterWalletSigOutcome` (upsert) |
| `PaymentDeposited` | `depositTime` + **refresh `maturityTime`** (see §4) |
| `ArbiterSet` (new) | `arbiter`, refresh `arbiterFeeBps` |
| `ArbiterPaid` (new, wallet) | arbiter payout (own field/entity, free choice) |

All other events (`StateChanged`, `Canceled`, `DisputeResolved`, `Withdrawn`, …)
are unchanged.

## 7. Deployment / reindex

1. Copy the v2 ABIs from this repo into the subgraph's `abis/`.
2. Update `subgraph.yaml`: new escrow contract **address + startBlock**
   (v2 is not deployed yet — placeholders TBD; the addresses in `src/config.ts`
   are still v1), new event signatures, new handlers.
3. `graph codegen && graph build`.
4. Deploy as a **new subgraph version** and let it fully reindex — the schema
   change plus the changed event topics make an in-place graft pointless
   (v1 and v2 are separate contract deployments anyway).
5. Keep the old version live until consumers are on SDK 3.x, then retire it.
